1. shop 컴포넌트 리랜더링 문제
[이슈 발생]
Link to 라우터를 생성한 후 연결된 컴포넌트가 상태변수를 전달받지 않아도
웹페이지가 새로 빌드되는 현상 발생! (마치 일반 a 태그처럼)

->  해결과정
(1) 전달하고있는 상태관리변수를 추가/제거 해보기 : 상관없이 리랜더링 확인 
(2) 콘솔로 상태확인 : 문제의 라우터 태그안에서 fn실행 및 console이 찍히지 않음. 
(3) path 주소를 바꾸어서 다른컴포넌트에 연결: 동일한 문제가 발생함을 확인

->> 해당 라우터 태그 자체에 문제가 생긴것을 인지

해당 a 태그에 적용 된 기능함수 중 
자식 요소 <a> 클릭시 부모 요소의 클래스가 이벤트버블링으로 인해 제거되지 않는 문제를 해결하기 위해 사용했던 e.stopPropagation(); 로 인해 라우터 리로드 문제를 발생시킴을 확인

해결방안: 
stopPropagation를 사용하여 이벤트 버블링을 막는 대신, 부모 요소를 직접 선택하여 클래스를 제거하는 방법으로 문제를 해결



2. shop 컴포넌트 상태관리 변수와 props 충돌
[이슈 발생]
 컴포넌트 밖 상단메뉴를 통해 shop 컴포넌트의 개별메뉴 진입시, 개별메뉴의 이름으로 props를 가지고 들어와 sMenu 상태관리변수의 값으로 할당하여 개별메뉴가 전환되도록 리랜더링 방식으로 구현 했으나, 상단메뉴를 통하지 않고 shop 컴포넌트 내부에서의 개별메뉴 클릭시 sMenu 상태관리변수를 변경해도 props값이 고정되어있어 (부모컴포넌트로부터 값을 받아올떄만 변경) 내부에서 메뉴전환이 되지 않는 문제가 발생.

->  해결과정
(1) 내부 개별메뉴 클릭시 props가 변경가능한지 확인했으나 불가 (내부에서 props는 읽기전용)
(2) props를 sMenu 상태변수의 초기값으로 지정: 최초로 상단메뉴 통해 개별메뉴 진입시 정상작동하나, 내부에서 메뉴전환이 일어나 sMenu가 setSmenu로 인해 갱신되면 더이상 상단메뉴를 통해 진입시 props를 상태변수의 초기값으로 쓰지않아 전환이 안되는 문제가 발생.

해결방안: 상단메뉴에서 개별메뉴의 이름으로 props를 받아와서 상태관리변수를 직접 바꾸려 하지 않고, 

 useEffect 구역에서 props의 값이 변경될떄만 내부 메뉴 버튼중 props의 값과 동일한 버튼만
 클릭이벤트를 발생시켜서 내부에서 개별메뉴 클릭한것과 같은 로직으로  상태관리변수가 전환되도록 처리하여 해결.\
